# CIP-0381: Plutus support for pairings over BLS12-381

CIP-0381 proposes an extension of the current Plutus builtin functions to provide
support for basic operations over the **BLS12-381** curve. This is the on-chain
primitive that makes cheap **pairing-based proof verification** (notably Groth16)
feasible inside a Plutus validator, and is therefore load-bearing for the
**Midnight → Cardano** direction of the recursive bridge.

## What it adds

Three new Plutus types are introduced, all representable as byte arrays:

- `bls12_381_G1_element`
- `bls12_381_G2_element`
- `bls12_381_MlResult` (Miller-loop result, a `GT`/target-group element)

A `bls12_381_G1_element` / `bls12_381_G2_element` can only be constructed from
byte arrays representing a point in the **prime-order subgroup** (deserialization
enforces subgroup membership), while an `MlResult` can only arise from the
`bls12_381_millerLoop` computation. Builtins cover group operations on G1/G2
(additive), MlResult (multiplicative), hash-to-curve for G1/G2, and the Miller
loop / final verification used to evaluate pairings.

## Why it matters for the bridge

A pairing is a bilinear map `e: G1 × G2 → GT` (bilinear, non-degenerate,
efficiently computable). Pairing computation is expensive, but pairings enable a
large class of primitives — including succinct-proof verification — that CIP-0381
lists as essential for **scaling and interoperability** (sidechains, bridges) of
Cardano. Because Groth16 verification reduces to a small fixed number of pairings,
CIP-0381's builtins let a Cardano validator verify a Groth16 proof within the
script budget. See the exec-cost budget analysis (to be added under
`/cardano/`) and the [source records](/sources/index.md).
