# CIP-0133 — Plutus multi-scalar multiplication over BLS12-381

CIP-0133 (status: Proposed) extends the Plutus builtins with efficient
**multi-scalar multiplication (MSM)** over the **BLS12-381** curve. Where
[CIP-0381](/standards/cip-0381.md) added the base curve/pairing operations that
make on-chain proof verification *possible*, CIP-0133 targets the operation that
makes it *affordable* — and is therefore directly load-bearing for the
**Midnight → Cardano** (Groth16) direction of the recursive bridge, where a
Groth16 proof must be verified inside a single Plutus transaction.

## What it adds

Two new Plutus functions are proposed, one per subgroup:

```hs
bls12_381_G1_multiScalarMul :: [Integer] -> [bls12_381_G1_element] -> bls12_381_G1_element
bls12_381_G2_multiScalarMul :: [Integer] -> [bls12_381_G2_element] -> bls12_381_G2_element
```

Given equal-length, non-empty arrays of scalars and group elements they compute
`M = Σ eᵢ · gᵢ` for the corresponding subgroup (empty or unequal inputs must
fail). The `bls12_381_G1_element` / `bls12_381_G2_element` types are reused from
CIP-0381, so these functions **naturally extend** the existing BLS12-381 builtin
set rather than introducing a new type surface. The implementation exposes the
`blst` library's optimized MSM (Pippenger-style), already a `cardano-base`
dependency.

## Why MSM matters for on-chain SNARK verification

MSM is the dominant cost in elliptic-curve SNARK verification and can become a
bottleneck in both proving and verification. In a **Groth16 verifier** the MSM
size scales with the **number of public inputs**; in a KZG-based PLONK verifier
it scales with the arithmetization. Doing MSM naively — even on top of CIP-0381's
BLS12-381 builtins — is prohibitively expensive on-chain:

- An MSM of just **10 G1 points** consumes **7.74%** of a transaction's
  computational budget.
- An MSM of size **> 129** points **cannot fit into a single transaction at
  all**, capping the circuit/public-input complexity a validator can verify.
- MSM cost grows **linearly** with size; `blst`'s optimized MSM beats a naive
  loop by an increasing margin (≈37× at 4000 points), and a naive MSM of size
  129 takes about as long as an optimized MSM of **> 4000** points — a proxy for
  the headroom a native builtin unlocks.

The net effect is lower verification cost and a much higher ceiling on
verifiable circuit size — precisely what a **trustless bridge** needs, since a
reliable trustless bridge requires SNARK proofs to efficiently attest to the
partner-chain (here, Midnight) state, and those proofs must clear the Plutus
exec budget to be verified on Cardano.
